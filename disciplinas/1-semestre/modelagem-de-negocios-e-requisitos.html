MODELAGEM DE NEGÓCIOS E REQUISITOS

Ementa

Estudo da dinâmica dos processos de negócio e das soluções de tecnologia da informação responsáveis pela operação das organizações. Compreensão das diversas fases envolvidas na Engenharia de Requisitos no contexto de desenvolvimento de software. Abordagem das técnicas e artefatos adequados para cada uma dessas fases.

Modelagem de Negócios

A modelagem de negócios visa representar os processos e a estrutura de funcionamento de uma organização. Ela permite visualizar, compreender e otimizar as operações, identificando oportunidades de melhoria. Para isso, utilizam-se ferramentas como BPMN (Business Process Model and Notation), que facilita a padronização da representação gráfica dos processos.

BPM e BPMN

BPM (Business Process Management) é uma abordagem para gerenciar os processos de negócios de uma organização. O BPM é um conjunto de práticas e tecnologias que permitem que uma empresa analise, modele, execute, monitore e melhore seus processos de negócios de forma contínua.

O BPMN (Business Process Model and Notation) é uma linguagem de modelagem gráfica padronizada, usada para representar visualmente os processos de negócios em diagramas. Ele é usado para modelar, analisar e melhorar os processos de negócios em um ambiente de BPM. O BPMN fornece um conjunto de símbolos e convenções gráficas para representar os processos de negócios de forma clara e precisa.

Bizagi

O Bizagi é uma ferramenta de modelagem de processos de negócio baseada na notação BPMN. Ele permite a criação de diagramas visuais intuitivos para representar fluxos de trabalho organizacionais, facilitando a comunicação entre as partes interessadas. Além disso, o Bizagi oferece funcionalidades para simulação e automação de processos, ajudando na identificação de gargalos e na otimização da eficiência operacional. Sua interface amigável e compatibilidade com padrões do mercado tornam-no uma escolha popular para empresas que desejam melhorar a gestão e execução de seus processos.

Engenharia de Requisitos

A engenharia de requisitos é a disciplina que define, documenta e gerencia as necessidades e expectativas dos stakeholders (partes interessadas) para um sistema de software. Esse processo envolve coleta, análise, especificação, validação e gestão dos requisitos ao longo do ciclo de vida do projeto. É o conjunto de tarefas e técnicas que conduzem a um entendimento dos requisitos e é essencial para garantir que os sistemas de software atendam aos requisitos dos usuários e aos objetivos do negócio.

Requisitos Funcionais e Não Funcionais

Os requisitos funcionais descrevem as funcionalidades e comportamentos que o sistema deve ter, como cadastro de usuários, envio de e-mails ou processamento de pagamentos. Já os requisitos não funcionais definem restrições e qualidades do sistema, como desempenho, segurança, escalabilidade e usabilidade.

Os requisitos funcionais definem o que o sistema deve fazer, enquanto os requisitos não funcionais determinam como o sistema deve operar. Ambos são essenciais para garantir que o sistema atenda tanto às necessidades do usuário quanto aos critérios de qualidade.

Stakeholder

Stakeholder é um termo que se refere a qualquer pessoa ou grupo que tenha interesse ou envolvimento em um projeto, empresa ou organização. Isso inclui indivíduos ou grupos internos e externos, como funcionários, clientes, acionistas, fornecedores, parceiros, comunidades locais e governos. É uma pessoa ou uma organização que tem influência direta ou indireta nos requisitos de um sistema, são provavelmente a maior fonte de requisitos. Requisitos incompletos podem ser resultado da negligência em relação a algum stakeholder.

Histórias de usuário

Histórias de usuário (user stories) é uma técnica utilizada em metodologias ágeis de desenvolvimento de software para descrever uma funcionalidade do sistema sob a perspectiva do usuário. Elas são escritas de forma simples e direta, geralmente em uma única frase, e contêm informações como o objetivo do usuário ao executar a funcionalidade e as vantagens que ele terá ao utilizá-la. As histórias de usuário são usadas para ajudar a equipe de desenvolvimento a entender as necessidades dos usuários e a desenvolver soluções que atendam às suas necessidades.
